# 你的数据，正在训练谁的模型？

你用 AI 助手越来越多。你问它今天天气、让它帮你写周报、叫它整理会议纪要。每一次对话，你都在留下数据。

你有没有想过一个问题：**这些数据，最后去了哪里？**

答案是：它们被用来训练了**别人的模型**。

你贡献了语料，优化了识别率，打磨了回复质量。但受益的不是你——是那个 AI 公司的下一个版本。

这个逻辑其实很奇怪。想象一下：你每天去健身房，教练记录你所有的训练数据、饮食、体脂——然后用这些数据出了一本畅销书，卖给别人。你本人呢？还是原来的你，没有任何变化。

这就是当前 AI 助手的悖论：**你用得越多，它们越强——但增强的不是「你的」助手，是「所有人的平均」助手。**

## 我们想要一个不同的答案

去年我们开始做一个叫「有数」的 Android 项目。初衷很简单：做一个 AI 助手客户端，但这个客户端的每一条数据，都属于你。

不是「声称」属于你——是**真的**属于你。

具体来说：

**对话记录存在你手机里。** 不是某个厂商的云端服务器。你想搜就搜，想导出就导出，想删就删。

**语音识别在手机本地跑。** 用的是 sherpa-onnx 框架，离线推理。你说的话不会上传到任何地方去做「云端转写」。

**每一条录音都是训练数据。** 但不是训练别人的模型——是训练**你自己的 ASR 模型**。你在手机上说了多少次「芒果」「周转筐」这种业务术语，你的模型就对这些词越来越敏感。这是一个越用越懂你的正向循环。

**AI 生成的文件也在你手机里。** 报告、PPT、图片——Agent 帮你生成的，全部归你。

这不是一个技术决定，这是一个**立场**。

## 技术视角 vs 用户视角

说回产品设计。有数里有一个设计我觉得值得展开聊聊：**用户能力 vs 技术技能的双层映射。**

什么意思？

在大多数 AI 客户端里，如果你想让 Agent 帮你做 PPT，你得选择「powerpoint 技能」。这是技术语言。

但在有数里，你只需要选择「帮我写作」这个**能力**。剩下的——选哪个技能、用哪个模型、怎么输出——全部由后端的一个叫 Skill Manager 的服务来决定。

这中间有一层映射：

```
「帮我写作」(用户语言)
      ↓
Skill Manager 映射
      ↓
writing-assistant (编排技能)
      ↓  clarify 选格式
      ↓  PPT / PDF / Word / Markdown
      ↓
powerpoint / claude-design / baoyu-infographic (技术技能)
```

为什么要绕这一层？

因为**用户不该学技术术语**。正常人的心智模型是「我要写个东西」，不是「我要调用 claude-design 技能并传入 PPT 格式参数」。用户语言到技术语言的翻译，应该由系统来做，不应该由用户心里默背。

更重要的是，这一层是**完全可配置的**。想加一个新格式？一条 API 调用搞定：

```
PUT /v1/capabilities/帮我写作/formats
{"format": "公众号长文", "skill": "baoyu-infographic"}
```

不用改 App 代码，不用发版。手机端的 Chip 列表自动更新。

这种设计背后是一个更根本的哲学：**把复杂度留在系统内部，把简单留给用户。**

## 然后呢？

有数现在是一个能用的客户端——语音识别、多轮对话、技能路由、手机工具集成、危险命令审批，这些都跑通了。

但我们不想把它做成一个「AI 平台」。我们想做的是**你自己的 AI 基地**。

你的数据，训练你的模型。你的偏好，塑造你的 Agent。手机上本地跑 ASR，将来本地跑小模型兜底。唯一需要联网的是你自建的 Hermes 服务器——端到端，不经过任何第三方。

**这不是反 AI。恰恰相反——这是让 AI 真正为你工作的唯一方式。**
